# CHANGELOG

## v0.0.10 (Unreleased)

- feat: **demand-driven ("pull") module resolution.** The eager dependency-graph BFS is replaced by a stateless `resolutionFrontier`: the plugin asks the core which modules are still MISSING to satisfy a host's directives, resolves + reads only those in a fixpoint loop, and does ZERO work for a directive-less file. Resolution now goes through the bundler's own `this.resolve` (aliases / `exports` maps / `resolve.extensions` / plugins) — bundler-faithful rather than a reimplementation — with a promise-cache so concurrent transforms share one resolve, and a fs-probe fast path kept only for the case a probe can't get wrong (a relative specifier with an explicit, transformable extension on disk). A correctness + ecosystem win, not only perf. The `donor` concept is renamed to `dependency` throughout (core / napi / wasm / plugin)
- feat: **full C++-style `@inline` directive surface.** `/* @inline */` on a call site now inlines that one call even when the callee is imported from another module (cross-file call-site inline); and a function whose *definition* is marked `@inline` is inlined at ALL its callers — including directive-less callers in other files ("mark once, inline everywhere") — via a build-start index of first-party `@inline` defs. Complements `@optimize` / `@flatten` / `@sroa` / `@unroll`
- fix: **shadowed named re-export miscompile.** A sourced named re-export (`export { x as y } from './s'`) whose target `./s` is out of scope no longer falls through to a shadowed `export *`, inlining the wrong module. The clause is authoritative: an unfollowable target stops resolution (matching the import-binding rule) instead of continuing to a lower-precedence same-named source. Fixes both the value (inline) and type (SROA) sides
- fix: **shadowed-callee miscompile.** The inliner matched a callee by name and would inline even when a caller's local/parameter shadowed the `@inline`/`@flatten` candidate (`f(add){ return add(1, 2) }` wrongly discarded the parameter and spliced the top-level `add`). A scope-precise shadow check now bails when the callee's leaf name is bound by an enclosing scope, across all inline modes (direct / block / call-site / cross-file)
- fix: **ambiguous `export *` miscompile.** A name provided by two independent `export *` re-exports — ambiguous in ES, since importing it is a SyntaxError — is no longer resolved to the first star arbitrarily; resolution returns nothing (a diamond that re-exports one underlying binding through two paths still resolves, the shared `visited` set collapsing it to a single provider)
- fix/hardening: **resolution, watch, and build safety.** The resolve cache is importer-directory-scoped (a bare specifier no longer mis-shares one importer's module with another under nested/duplicated deps or workspace hoisting); a native panic on unexpected input now warns and passes the module through UNoptimized rather than failing the user's build; `watchChange` invalidates the stat / resolve caches so an add / delete / rename can't leave a stale resolution; a file imported under multiple specifiers inlines from all of them; unknown frontier kinds are skipped rather than mis-routed to value resolution; in-scope source dirs whose name starts with `.` (e.g. `.generated/`) are indexed; and package type resolution consults the DefinitelyTyped `@types/<pkg>` package when a package ships no own declarations (`typesVersions` remains a known gap)
- feat: cross-module inlining follows a **re-exported imported binding**
  — the value-side analogue of v0.0.9's type re-export following. A module that re-surfaces another module's function under its own exported name (`import { set as set$1 } from './vec4.js'; const set = set$1; export { set }` — exactly how `mathcat`'s `quat` re-uses `vec4`'s componentwise ops) now inlines into an `@optimize`/`@flatten` host. Previously the call survived (the inliner only accepted an arrow/function-expression initializer, not an identifier aliasing an imported callable), pinning any scratch buffer it wrote as a module array and blocking SROA. The core resolves the const-alias → import → real definition (`resolve_value_origin`, sharing the dependency-graph hop primitives + `(dependency,name)` memo with the type resolver), and the plugin's dependency resolution follows a dependency's plain import when that import feeds a re-exported binding (so the defining submodule is gathered). Resolution follows JS binding precedence — a local declaration or explicit import/named-export **shadows** `export *`, and a higher-priority binding that can't be followed to a callable stops resolution rather than falling through to a shadowed same-named wildcard export (which would inline the wrong function)
- fix: type re-export resolution (`resolve_type_alias_shape`, from v0.0.9) now respects the same shadowing precedence. A module that declares or explicitly re-exports a type `X` which isn't a scalarizable shape (e.g. `export type X = (n: number) => number`) while also doing `export * from './other'` (where `./other` has a tuple `X`) previously fell through to the wildcard's tuple and fired type-directed SROA on a value the authoritative local `X` says is not a tuple — a miscompile. A locally-declared or explicitly-re-exported type is now authoritative: if it doesn't resolve to a scalarizable shape, resolution stops instead of falling through to a shadowed `export *`
- fix: cross-file inlining no longer corrupts the consumer's sourcemap. Dependency AST spliced into a consumer kept its *dependency* source spans, which index the dependency's source text; against the consumer sourcemap those are out of range (oxc codegen `debug_assert`s on a named node past the consumer length — which blocked debug-build testing — and silently misplaced the mapping in release). Dependency material (candidate bodies, forwarded imports, hoisted decls) now has its spans neutralised before codegen when a sourcemap is emitted, so inlined code maps to the top of the consumer rather than to a bogus offset

## v0.0.9

- feat: cross-package `.d.ts` type resolution
  — the type-shape oracle resolves imported type aliases from a package's declaration surface (package.json `exports` `"types"` condition -> `types`/`typings` -> sibling `.d.ts`) and follows the `.d.ts` re-export graph (bare `export * from`, named / `export type` re-exports; namespace `export * as ns` excluded). Type-directed passes (SROA, module-scratch localization) now fire on types imported from published packages that ship a `.js` + `.d.ts` split (e.g. `mathcat`'s `Vec3`/`Quat`), not only local/relative imports
- feat: removed the `MAX_DONORS` donor cap — donor gathering is bounded by the finite reachable graph (`seen` dedup + `inScope`), like a bundler, instead of a silent per-consumer count. Cross-module optimization no longer quietly degrades past a magic number of imports
- fix: `resolve_type_alias_shape` memoises `(donor, name)` — a densely cross-re-exporting `.d.ts` graph no longer re-expands exponentially (previously a multi-minute hang); `exports` wildcard subpaths resolve longest-match-wins; `watchChange` invalidates the package-entry caches on a dependency's `package.json` edit
